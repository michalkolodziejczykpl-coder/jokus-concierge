# send-sms — Supabase Send SMS Hook → SMSAPI.pl

Delivers phone OTP codes (registration step 3) via **SMSAPI.pl** instead of
Twilio. Supabase has no native SMSAPI provider, so we register this Edge
Function as a **Send SMS Hook**: Supabase signs each OTP request and POSTs it
here; we verify the signature and forward the code through the SMSAPI REST API.

The application code does **not** change — `signInWithOtp` / `updateUser({ phone })`
/ `verifyOtp` stay exactly as they are. Only the SMS delivery channel changes.

## How it works

1. App calls `supabase.auth.updateUser({ phone })` → Supabase needs to send an SMS.
2. Supabase POSTs to this function (Standard Webhooks signed) with:
   ```json
   { "user": { "phone": "+48123456789" }, "sms": { "otp": "123456" } }
   ```
3. We verify the signature with `SEND_SMS_HOOK_SECRET`, then call
   `https://api.smsapi.pl/sms.do` with `to=48123456789`, the Polish message, and
   the approved sender (`from`).
4. Return `200 {}` on success. Bad signature → `401` (no SMS sent). SMSAPI error
   → `500` (logged).

## Function secrets (never commit)

| Secret                  | Where it comes from                                              |
| ----------------------- | --------------------------------------------------------------- |
| `SMSAPI_TOKEN`          | SMSAPI.pl panel → API → OAuth token                             |
| `SMSAPI_SENDER`         | Approved sender field (e.g. `JOKUS`; max 11 chars)              |
| `SEND_SMS_HOOK_SECRET`  | Generated by Supabase when you enable the Send SMS hook (step 4) |

## Owner deployment steps (Michał)

1. **SMSAPI token + sender.** SMSAPI.pl panel → API → generate an OAuth token →
   that's `SMSAPI_TOKEN`. Add/approve a sender field "JOKUS" (Pola nadawcy) →
   that's `SMSAPI_SENDER`. If no approved sender yet, use the account's default /
   "Test" eco sender.

2. **Deploy the function** (the hook calls it with its own signature, not a user
   JWT — so disable JWT verification):
   ```bash
   supabase functions deploy send-sms --no-verify-jwt
   ```

3. **Set the SMSAPI secrets:**
   ```bash
   supabase secrets set SMSAPI_TOKEN=... SMSAPI_SENDER=JOKUS
   ```

4. **Enable the hook.** Supabase Dashboard → Authentication → Hooks →
   **Send SMS hook** → point it at
   `https://uveeqjidyuumcddnfnop.supabase.co/functions/v1/send-sms`, then copy the
   generated **secret** (`v1,whsec_...`) and set it:
   ```bash
   supabase secrets set SEND_SMS_HOOK_SECRET=v1,whsec_...
   ```
   Redeploy the function if it was already running before the secret existed:
   ```bash
   supabase functions deploy send-sms --no-verify-jwt
   ```
   (Also make sure Authentication → Sign In / Providers → **Phone** is enabled so
   Supabase actually runs the OTP flow.)

5. **Turn on the app flow.** In Vercel set `PHONE_OTP_ENABLED=true` and redeploy.
   Registration step 3 will now send a real code.

## Verifying

- E2E: register by email → `/rejestracja/uzupelnij` → enter your phone → receive
  the `Twój kod JOKUS: …` SMS → enter it → `phone_verified` is flipped
  server-side → you reach the app.
- Bad signature: a POST with a wrong/missing `webhook-signature` returns `401`
  and sends no SMS.

## Notes

- `to` is sent without the leading `+` (SMSAPI wants `48XXXXXXXXX`); we strip all
  non-digits from `user.phone`.
- SMSAPI returns HTTP 200 even on logical errors, signalling failure via an
  `error` code in the JSON body — we check the body, not just the HTTP status.
